# Выбор и настройка мониторинга в системе

## Мотивация

Сейчас компания уже сталкивается с серьёзными проблемами: теряются или зависают заказы, растёт нагрузка после открытия API, медленно работает MES, увеличивается число жалоб от клиентов и партнёров.
Одной Яндекс Метрики для такой системы уже недостаточно, потому что она показывает в основном поведение пользователей сайта, но не даёт полной картины по backend-сервисам, очередям, базам данных и внутренним интеграциям.

Добавление мониторинга нужно для того, чтобы:

- быстрее находить причины сбоев и деградации
- видеть, где именно теряются или застревают заказы
- контролировать нагрузку на Shop API, CRM API, MES API, RabbitMQ и базы данных
- отслеживать последствия открытия внешнего API
- сократить время реакции на инциденты
- понять, когда системе уже не хватает текущих ресурсов

Для бизнеса это даёт понятную выгоду: меньше потерянных заказов, меньше просрочек, меньше жалоб и меньше риск потери контрактов.

---

## Выбор подхода к мониторингу

Для системы я бы использовал несколько подходов.

### 1. RED для API
Для Shop API, CRM API и MES API лучше всего подходит подход RED:
- Rate — сколько запросов приходит
- Errors — сколько ошибок возникает
- Duration — насколько долго обрабатываются запросы

### 2. USE для инфраструктуры и баз данных
Для серверов приложений, RabbitMQ и баз данных лучше использовать USE:
- Utilization — насколько ресурс загружен;
- Saturation — есть ли очередь ожидания или исчерпание ресурса;
- Errors — есть ли ошибки при работе ресурса.

### 3. Дополнительно — бизнесовые метрики
Кроме технических метрик, для этой системы обязательно нужны бизнесовые и процессные метрики, потому что главная проблема компании — не просто ошибки API, а потерянные и просроченные заказы.

## Метрики, которые нужно отслеживать

## 1. API-метрики по подходу RED

### Number of requests (RPS) for internet shop API
**Зачем нужна:** показывает общую нагрузку на API интернет-магазина. Помогает понять, как меняется поток заказов и когда система выходит на новый уровень нагрузки.
**Ярлыки:** `method`, `route`, `status_code`, `environment`.

### Number of requests (RPS) for CRM API
**Зачем нужна:** показывает нагрузку на CRM, через которую подтверждаются и закрываются заказы.
**Ярлыки:** `method`, `route`, `status_code`, `environment`.

### Number of requests (RPS) for MES API
**Зачем нужна:** особенно важна, так как MES участвует в расчёте стоимости и в работе операторов.
**Ярлыки:** `method`, `route`, `status_code`, `environment`.

### Number of requests (RPS) per user for internet shop API
**Зачем нужна:** позволяет выявлять аномальное поведение клиентов или интеграций, а также потенциальную перегрузку со стороны отдельных пользователей.
**Ярлыки:** `user_id` или `client_id`, `route`, `environment`.
Для внешнего API лучше использовать не обычный user_id, а идентификатор клиента или партнёра.

### Number of requests (RPS) per user for CRM API
**Зачем нужна:** позволяет понять активность продавцов и выявлять аномальную нагрузку или неудачную автоматизацию.
**Ярлыки:** `user_id`, `route`, `environment`.

### Number of requests (RPS) per user for MES API
**Зачем нужна:** помогает увидеть, как работают операторы и не создаёт ли кто-то чрезмерную нагрузку на MES.
**Ярлыки:** `user_id`, `route`, `environment`.

### Response time (latency) for shop API
**Зачем нужна:** показывает, насколько быстро отвечает магазин, и позволяет увидеть деградацию пользовательского сценария.
**Ярлыки:** `route`, `method`, `status_code`, `environment`.
Лучше хранить как histogram и смотреть p50/p95/p99.

### Response time (latency) for CRM API
**Зачем нужна:** позволяет контролировать скорость работы внутренних процессов продавцов.
**Ярлыки:** `route`, `method`, `status_code`, `environment`.

### Response time (latency) for MES API
**Зачем нужна:** одна из ключевых метрик, так как именно MES сейчас является одним из узких мест.
**Ярлыки:** `route`, `method`, `status_code`, `environment`.

### Number of HTTP 200 for shop API / CRM API / MES API
**Зачем нужна:** показывает число успешных запросов и помогает считать долю успешной обработки.
**Ярлыки:** `route`, `method`, `environment`.

### Number of HTTP 500 for shop API / CRM API / MES API
**Зачем нужна:** показывает внутренние ошибки приложений. Рост этой метрики — признак инцидента или деградации.
**Ярлыки:** `route`, `method`, `environment`.

### Number of simultanious sessions for shop API / CRM API / MES API
**Зачем нужна:** помогает оценить текущую одновременную нагрузку на приложения.
**Ярлыки:** `environment`, `client_type`.

### Kb transferred (received) / Kb provided (sent) for shop API / CRM API / MES API
**Зачем нужна:** полезно для оценки сетевой нагрузки, особенно для сценариев загрузки моделей и интеграционных вызовов.
**Ярлыки:** `route`, `environment`.

## 2. Инфраструктурные метрики по подходу USE

### CPU % for shop API / CRM API / MES API
**Зачем нужна:** показывает, насколько приложение упирается в процессор. Особенно важно для MES API.
**Ярлыки:** `instance`, `environment`.

### Memory Utilisation for shop API / CRM API / MES API
**Зачем нужна:** помогает отслеживать утечки памяти, рост потребления и риски OOM.
**Ярлыки:** `instance`, `environment`.

### Memory Utilisation for shop db instance / MES db instance
**Зачем нужна:** высокая загрузка памяти базы может говорить о нехватке ресурсов или неэффективных запросах.
**Ярлыки:** `db_instance`, `environment`.

### Number of connections for shop db instance / MES db instance
**Зачем нужна:** показывает, не упирается ли система в лимиты соединений и не слишком ли растёт нагрузка на БД.
**Ярлыки:** `db_instance`, `environment`.

### Size of shop db instance / Size of MES db instance
**Зачем нужна:** помогает планировать ресурсы и видеть аномальный рост данных.
**Ярлыки:** `db_instance`, `environment`.

### Size of S3 storage
**Зачем нужна:** важно для контроля роста 3D-файлов и прогноза расходов на хранение.
**Ярлыки:** `bucket`, `environment`.

## 3. Метрики RabbitMQ

### Number of dead-letter-exchange letters in RabbitMQ
**Зачем нужна:** показывает, сколько сообщений не удалось обработать нормально. Рост этой метрики напрямую связан с потерей или зависанием заказов.
**Ярлыки:** `queue`, `exchange`, `routing_key`, `environment`.

### Number of message in flight in RabbitMQ
**Зачем нужна:** показывает, сколько сообщений находится в обработке. Если метрика растёт и не снижается, значит система не успевает обрабатывать поток.
**Ярлыки:** `queue`, `consumer`, `environment`.

## План действий

### 1. Развернуть стек мониторинга
- Создать инстанс Prometheus для сбора метрик
- Настроить Grafana для визуализации
- Настроить централизованное логирование через ELK
- Добавить систему трассировки OpenTelemetry + Jaeger

### 2. Добавить экспортёры и сбор метрик
- Подключить сбор метрик для Java Spring Boot приложений
- Подключить сбор метрик для C# приложения MES
- Настроить сбор системных метрик с инстансов
- Подключить exporter для RabbitMQ
- Подключить exporter для PostgreSQL / managed DB
- Подключить мониторинг S3-хранилища

### 3. Добавить прикладные метрики в код
- Метрики по статусам заказов
- Метрики по времени расчёта стоимости
- Метрики по времени нахождения заказа в статусе
- Метрики по числу зависших заказов
- Метрики по retry и ошибкам обработки сообщений

### 4. Настроить дашборды
Нужно сделать отдельные дашборды:
- общий health dashboard
- dashboard по API
- dashboard по RabbitMQ
- dashboard по БД
- dashboard по MES
- бизнесовый dashboard по заказам и статусам

### 5. Настроить алерты
Алерты нужны на:
- рост HTTP 500
- рост latency p95/p99
- рост очередей RabbitMQ
- появление сообщений в DLQ
- превышение времени расчёта стоимости
- зависшие заказы
- нехватку CPU, памяти, соединений с БД
