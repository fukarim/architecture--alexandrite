# Архитектурное решение по логированию

## Мотивация

Сейчас проблемы в системе часто разбираются со слов клиента или сотрудника.
Это сильно замедляет расследование: поддержка не видит, что именно происходило внутри сервисов, а разработчикам приходится тратить много времени на восстановление цепочки событий.

Логирование нужно добавить, чтобы:
- быстрее понимать, что произошло внутри системы
- видеть ошибки и нестандартные ситуации без обращения к клиенту
- упростить разбор проблем с заказами
- снизить нагрузку на поддержк

Для бизнеса это даст:
- меньше времени на расследование проблем
- меньше потерянных и зависших заказов
- быстрее обработку обращений клиентов
- снижение нагрузки на поддержку и разработку
- меньше риск повторения одних и тех же инцидентов

### На какие метрики и показатели повлияет внедрение логирования
1. Сократится время диагностики и устранения инцидентов.
2. Количество обращений в поддержку по неясным проблемам уменьшится.
3. Количество потерянных или зависших заказов станет проще находить и устранять причины.
4. Время ответа клиенту по проблемному заказу сократится.

## Какие системы нужно покрыть логированием

Логи нужно собирать из всех систем, через которые проходит заказ или где может возникнуть ошибка.

### 1. Интернет-магазин
Нужно логировать:
- создание заказа
- загрузку 3D-модели
- ошибки загрузки
- обращения к S3
- отправку заказа дальше по процессу
- ошибки API и интеграций

### 2. CRM
Нужно логировать:
- получение заказа
- изменение статуса
- подтверждение заказа
- закрытие заказа
- ошибки обработки и интеграций

### 3. MES
Нужно логировать:
- получение заказа
- запуск и завершение расчёта стоимости
- длительное выполнение расчёта
- смену статусов заказа
- действия операторов
- ошибки чтения, записи и интеграций

### 4. RabbitMQ
Нужно логировать:
- публикацию сообщения
- получение сообщения
- повторную обработку
- ошибки обработки

### 5. S3-хранилище
Нужно логировать:
- загрузку файла
- чтение файла
- ошибки доступа
- отсутствие нужного файла

### 6. Системные и инфраструктурные логи
Нужно собирать:
- системные логи инстансов
- логи приложений
- логи ошибок базы данных

## Какие логи нужны с уровнем INFO

На уровне **INFO** нужно писать только важные бизнесовые и технические события, которые помогают восстановить путь заказа, но не создают лишний шум.

### Интернет-магазин
- заказ создан
- 3D-файл загружен
- файл сохранён в S3
- заказ отправлен в MES или в очередь
- получен успешный ответ по API

### CRM
- заказ получен
- заказ подтверждён
- статус заказа изменён
- заказ закрыт

### MES
- заказ получен на расчёт
- расчёт стоимости запущен
- расчёт стоимости завершён
- стоимость рассчитана
- заказ переведён в новый статус
- оператор взял заказ в работу
- заказ передан на упаковку
- заказ отправлен

### RabbitMQ
- сообщение опубликовано
- сообщение получено consumer’ом
- сообщение успешно обработано
- сообщение отправлено на повторную обработку

### S3
- файл успешно загружен
- файл прочитан
- файл не найден.

### Какие поля должны быть в INFO-логах
Для всех ключевых логов желательно использовать единый формат и передавать:
- `timestamp`
- `service_name`
- `environment`
- `order_id`
- `trace_id`
- `client_id` или `partner_id`
- `user_id`
- `operation`
- `status_before`
- `status_after`
- `result`

## Другие уровни логирования

### WARN
Используется для подозрительных, но не критичных ситуаций:
- слишком долгий расчёт стоимости
- повторная обработка сообщения
- рост времени ответа
- попытка обратиться к несуществующему файлу

### ERROR
Используется для ошибок, которые мешают выполнению операции:
- не удалось сохранить или прочитать файл
- ошибка обработки заказа
- не удалось отправить или обработать сообщение
- ошибка при смене статуса
- ошибка обращения к базе или внешнему сервису

### FATAL
Используется редко, когда сервис не может продолжать работу:
- невозможность поднять приложение
- отказ соединения с обязательным компонентом при старте

## Что настраивать в первую очередь

Так как команда не сможет сразу покрыть всё, логирование и трейсинг в первую очередь нужно внедрять в самых критичных системах.

### 1. MES
Это главный кандидат, потому что:
- здесь считается стоимость
- здесь возникают задержки
- через MES проходит большой объём ключевых событий заказа

### 2. RabbitMQ
Это вторая по важности точка, потому что:
- сообщения могут теряться
- возможны ретраи
- именно здесь заказ может пропасть между системами

### 3. Интернет-магазин
Важно быстро видеть:
- создание заказа
- загрузку файла
- отправку заказа дальше
- ошибки внешнего API

### 4. CRM
CRM можно подключать следующим этапом, когда будут покрыты основные проблемные места.

## Предлагаемое решение

### Общая идея

Предлагается внедрить централизованное логирование на базе ELK-стека:

- Filebeat агент сбора логов на инстансах
- Logstash для обработки и нормализации
- Elasticsearch для хранения и поиска
- Kibana для анализа логов

Такой подход хорошо сочетается с уже выбранным стеком:
- Prometheus + Grafana для мониторинга
- OpenTelemetry + Jaeger для трейсинга
- ELK для логов

### Как будет работать решение

1. Приложения пишут структурированные JSON-логи.
2. Логи с инстансов собираются агентом.
3. Агент отправляет логи в Logstash.
4. Logstash:
   - парсит логи
   - нормализует поля
   - обогащает данные
   - отправляет их в Elasticsearch
5. В Kibana можно искать логи по:
   - `order_id`
   - `trace_id`
   - `service_name`
   - `level`
   - `client_id`
   - `partner_id`
   - `environment`

### Что нужно доработать в приложениях

- перейти на единый формат логов JSON
- добавить обязательные поля:
  - `timestamp`
  - `level`
  - `service_name`
  - `environment`
  - `order_id`
  - `trace_id`
  - `client_id` / `partner_id`
  - `user_id`
  - `operation`
  - `message`
- добавить логирование ключевых бизнес-событий
- связать логи с трейсингом через `trace_id`

## Безопасность

Система логирования будет содержать чувствительную техническую информацию, поэтому нужна политика безопасности.

### Доступ
- доступ к Kibana только для сотрудников компании
- аутентификация через корпоративную учётную запись
- ролевой доступ:
  - DevOps / SRE
  - backend-разработчики
  - тимлид
  - поддержка с ограниченным доступом
- внешний доступ из интернета запрещён
- доступ только из внутренней сети или через VPN

### Работа с чувствительными данными
В логи нельзя писать:
- персональные данные клиентов в открытом виде
- полные payload’ы сообщений
- токены, пароли, ключи доступа

---

## Политика хранения логов

Для логов лучше использовать отдельные индексы по системам, например:
- `shop-logs-*`
- `crm-logs-*`
- `mes-logs-*`
- `rabbitmq-logs-*`
- `infra-logs-*`

### Срок хранения
- INFO / WARN / ERROR для приложений — 30 дней
- инфраструктурные логи — 30 дней
- логи безопасности и аудита — 90 дней

### Размер и контроль роста
- настроить rollover индексов
- ограничить размер индекса, например по дневной или недельной ротации
- старые индексы переводить в более дешёвое хранение или удалять

## Как превратить систему сбора логов в систему анализа

### Алертинг
Нужно настроить алерты на:
- резкий рост ERROR-логов
- повторяющиеся ошибки обработки заказов
- рост сообщений о retry
- ошибки чтения или записи в S3
- всплеск 5xx по конкретному сервису
- рост числа ошибок расчёта стоимости

### Поиск аномалий
Нужно искать аномалии, например:
- резкий рост числа созданных заказов за короткое время
- резкий рост количества запросов от одного клиента или партнёра
- резкий рост WARN/ERROR по одному роуту
- большое количество повторных обработок сообщений
- нетипичный рост логов отказов авторизации или ошибок API

Это полезно для:
- раннего обнаружения DDoS
- поиска неудачной интеграции партнёра
- обнаружения массового сбоя после релиза
