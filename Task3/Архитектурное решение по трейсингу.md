# Архитектурное решение по трейсингу

## Мотивация

В системе «Александрит» заказ проходит через несколько компонентов: интернет-магазин, MES, CRM, RabbitMQ, S3-хранилище и внешний API для партнёров.
Сейчас одна из главных проблем компании в том, что заказы могут зависать, теряться или оставаться в непонятном состоянии.
Обычный мониторинг помогает увидеть рост ошибок, задержек и очередей, но не позволяет понять путь конкретного заказа через все сервисы.

Поэтому в дополнение к мониторингу нужно внедрить трейсинг. Он позволит:
- видеть полный путь заказа по системе
- понимать, на каком сервисе или интеграции заказ завис
- находить проблемы с передачей сообщений через RabbitMQ
- быстрее расследовать инциденты
- связывать технические сбои с конкретными заказами и статусами

### На какие метрики и показатели повлияет внедрение трейсинга
1. Среднее время диагностики и устранения инцидентов
2. Количество зависших заказов — станет проще выявлять место, где заказ остановился
3. Время прохождения заказа по статусам — можно будет точнее определять этап задержки
4. Процент успешно завершённых заказов— должен вырасти за счёт более быстрой диагностики проблем
5. Количество обращений в поддержку по потерянным заказам — должно уменьшиться

## Какие системы нужно покрыть трейсингом

Трейсингом нужно покрыть все компоненты, через которые проходит заказ или связанные с ним сообщения.

### 1. Интернет-магазин
Нужно трассировать:
- создание заказа
- загрузку 3D-модели
- обращение к S3
- отправку заказа дальше по процессу
- вызовы внутренних сервисов и публикацию сообщений

### 2. MES
Нужно трассировать:
- получение заказа на расчёт
- сам расчёт стоимости
- изменение статусов заказа
- действия операторов
- отправку сообщений в CRM и другие компоненты

### 3. CRM
Нужно трассировать:
- получение заказа
- подтверждение заказа
- смену статусов
- закрытие заказа

### 4. RabbitMQ
Нужно трассировать:
- публикацию сообщений
- получение сообщений consumer’ами
- ретраи
- ошибки обработки

### 5. S3-хранилище
Нужно трассировать:
- загрузку файла
- чтение файла
- ошибки доступа к файлу

### 6. Внешний API для партнёров
Нужно трассировать:
- входящий запрос от партнёра
- создание заказа через API
- дальнейшее прохождение заказа по внутренней цепочке

## Где заказ может сломаться или зависнуть

Основные точки риска:
1. При создании заказа в интернет-магазине
2. При загрузке или чтении 3D-файла из S3
3. При передаче заказа в MES
4. Во время долгого расчёта стоимости в MES
5. При публикации сообщения в RabbitMQ
6. При обработке сообщения consumer’ом
7. При смене статуса заказа в MES или CRM
8. При создании заказа через внешний API партнёров
9. При закрытии заказа в финале бизнес-процесса

## Какие данные должны попадать в трейсинг

Для каждого trace и span нужно сохранять данные, которые позволят связать техническое событие с конкретным заказом.

### Обязательные данные

- `trace_id`
- `span_id`
- `parent_span_id`
- `order_id`
- `service_name`
- `operation_name`
- `timestamp_start`
- `timestamp_end`
- `duration_ms`
- `status`
- `error_code`
- `environment`

### Бизнесовые и прикладные атрибуты
- `client_id` или `partner_id`
- `user_id`
- `order_status_before`
- `order_status_after`

### Для RabbitMQ
- `queue`
- `exchange`
- `routing_key`
- `consumer`
- `retry_count`

### Для HTTP
- `http_method`
- `route`
- `http_status_code`

### Для S3 и файлов
- `file_id` или `model_id`
- тип операции: upload / read / delete

## Предлагаемое решение

### Общая идея

Предлагается внедрить трейсинг на базе OpenTelemetry.
Трейсы будут собираться из приложений и передаваться через OpenTelemetry Collector в Jaeger.
Такое решение хорошо согласуется с уже выбранным стеком мониторинга: Prometheus + Grafana + ELK + OpenTelemetry + Jaeger.

### Технологии

OpenTelemetry SDK Shop / CRM / MES API

OpenTelemetry Collector - для приёма, обработки и маршрутизации трейсов

Jaeger - для хранения и просмотра распределённых трейсов

### Как будет работать решение

1. При входящем HTTP-запросе в Shop API, CRM API или MES API создаётся trace.
2. Контекст трейсинга передаётся дальше:
   - через HTTP между сервисами
   - через headers сообщений в RabbitMQ
   - при работе с S3
   - при запросах к базе данных
3. На каждом этапе создаётся span:
   - создание заказа
   - загрузка модели
   - расчёт стоимости
   - публикация сообщения
   - обработка сообщения
   - смена статуса
   - подтверждение и закрытие заказа
4. Все сервисы отправляют трейсы в OpenTelemetry Collector.
5. Collector передаёт их в Jaeger.
6. Команда сможет искать путь заказа по `order_id`, `trace_id`, `partner_id`, `client_type` и статусам заказа.

### Что нужно доработать в приложениях

- добавить OpenTelemetry instrumentation в Shop API, CRM API и MES API
- унифицировать использование `order_id` как ключевого бизнесового идентификатора
- пробрасывать trace context через RabbitMQ headers
- добавлять span attributes:
  - `order_id`
  - `client_id` / `partner_id`
  - `order_status_before`
  - `order_status_after`
  - `queue`
  - `routing_key`
  - `environment`
- инструментировать обращения к S3 и БД
- связать трейсинг с логами через общий correlation id

## Компромиссы

### 1. Трейсинг не заменяет мониторинг
Трейсинг нужен вместе с метриками и логами. Сам по себе он не покажет полную картину нагрузки или насыщенности системы.

### 2. Не весь путь заказа можно покрыть одинаково глубоко
Лучше всего покрываются собственные backend-сервисы. Если часть логики находится во внешней системе или сложном компоненте, trace может обрываться.

### 3. Есть накладные расходы
Полный трейсинг всех операций увеличивает нагрузку на приложения и хранилище трейсов. Поэтому сначала разумно покрывать только критичные сценарии заказа.

### 4. Нужна дисциплина передачи контекста
Если хотя бы один сервис не будет пробрасывать `trace_id` и `order_id`, то полный путь заказа станет неполным.

## Безопасность

Система трейсинга будет содержать чувствительную техническую информацию, поэтому нужно предусмотреть защиту доступа.

### Доступ внутри компании
- доступ к Jaeger только для сотрудников компании
- аутентификация через корпоративную учётную запись
- ролевой доступ:
  - DevOps / SRE
  - backend-разработчики
  - тимлид
  - поддержка и администраторы при необходимости
- доступ только из внутренней сети или через VPN

### Доступ снаружи
- Jaeger и OpenTelemetry Collector не должны быть доступны напрямую из интернета
- внешний доступ к интерфейсу запрещён

### Защита данных
- не передавать персональные данные клиентов в span attributes
- не сохранять полные payload сообщений
- использовать только технические идентификаторы:
  - `order_id`
  - `client_id`
  - `partner_id`
  - `user_id`
- маскировать чувствительные сообщения об ошибках
