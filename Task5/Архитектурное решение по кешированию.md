# Архитектурное решение по кешированию

## Мотивация

Сейчас у компании есть две заметные проблемы с производительностью:

1. MES медленно открывает страницу со списком заказов, хотя уже были добавлены фильтры и пагинация.
2. Клиенты долго ждут выполнение заказа, особенно на этапе расчёта стоимости.

Полностью решить вторую проблему только кешированием нельзя, потому что основная причина там — долгий расчёт стоимости в MES.
Это скорее задача асинхронной обработки и оптимизации самого процесса расчёта.

При этом кеширование хорошо подходит для ускорения часто читаемых данных, в первую очередь в MES.
Поэтому его имеет смысл внедрять там, где много повторных чтений и сравнительно небольшая чувствительность к задержке в несколько секунд.

Кеширование должно решить следующие проблемы:
- снизить нагрузку на MES DB
- ускорить открытие страницы со списком заказов
- ускорить получение агрегатов по статусам
- уменьшить количество одинаковых запросов к базе
- повысить отзывчивость MES для операторов

### Что планируется включить в кеширование

Предлагается кешировать:
- список заказов для главной страницы MES
- агрегаты по статусам заказов для dashboard
- карточку заказа при повторном чтении
- справочные данные, если они используются часто и редко меняются

---

## Какую часть системы имеет смысл кешировать

### 1. Список заказов для главной страницы MES
Это главный кандидат на кеширование, потому что:
- операторы часто открывают эту страницу
- запросы на чтение повторяются
- чтение происходит чаще, чем массовое изменение данных

### 2. Агрегаты по статусам
Например:
- количество заказов в `PRICE_CALCULATED`;
- количество заказов в `MANUFACTURING_STARTED`;
- количество заказов в `PACKAGING`.

Такие данные хорошо подходят для кеширования, потому что они часто используются в интерфейсе и не требуют обновления с точностью до миллисекунды.

### 3. Карточка заказа
Если один и тот же заказ открывают несколько раз подряд, повторное чтение из базы избыточно.

### 4. Справочные данные
Например:
- набор статусов
- настройки отображения
- справочники, редко изменяющиеся в системе

## Предлагаемое решение

### Выбор типа кеширования

Я предлагаю использовать серверное кеширование на стороне MES API.

### Почему серверное кеширование
Серверное кеширование подходит лучше, потому что:
- позволяет снизить нагрузку на базу данных
- работает одинаково для всех операторов
- легче централизованно управлять инвалидацией
- помогает именно в том месте, где сейчас есть узкое место — на backend и в БД.

### Почему не клиентское кеширование
Клиентское кеширование здесь менее эффективно, потому что:
- операторам важны актуальные новые заказы
- данные быстро меняются
- один и тот же список используется разными пользователями
- нужен единый механизм управления свежестью данных

Поэтому основной эффект даст именно кеш на стороне сервиса.

---

## Технология

Для кеша предлагается использовать Redis, потому что он:
- быстрый
- подходит для read-heavy сценариев
- поддерживает TTL
- позволяет гибко удалять ключи
- может использоваться несколькими инстансами MES API

---

## Паттерн кеширования

Для этого сценария лучше всего подходит паттерн Cache-Aside.

### Почему выбран Cache-Aside

Схема работы:
1. MES API сначала ищет данные в кеше.
2. Если данные есть — возвращает их из Redis.
3. Если данных нет — читает их из MES DB, кладёт в Redis и возвращает клиенту.

Этот паттерн лучше всего подходит, потому что:
- основная проблема в системе связана с тяжёлым чтением
- не нужно заранее класть в кеш все данные
- приложение может гибко управлять тем, что именно кешировать
- проще реализовать инвалидацию при изменении статуса заказа

### Почему не Write-Through
Write-Through означает, что при записи данные сразу сохраняются и в БД, и в кеш.

Этот паттерн здесь хуже, потому что:
- текущая проблема в системе связана в первую очередь не с записью, а с чтением
- основной кешируемый объект — не только карточка заказа, но и списки, и агрегаты
- обновление списков и агрегатов при каждом изменении статуса усложнит реализацию

### Почему не Refresh-Ahead
Refresh-Ahead предполагает, что система заранее обновляет кеш до истечения срока его жизни.

Этот паттерн может быть полезен для некоторых горячих данных, но как основной вариант он хуже, потому что:
- не все фильтры и страницы можно заранее предсказать
- можно тратить ресурсы на прогрев ненужных ключей
- реализация получается сложнее

Поэтому основным паттерном предлагается сделать Cache-Aside, а прогрев кеша можно рассматривать как дополнительную оптимизацию позже.

## Сущности, участвующие в кешировании

В процессе участвуют:
- MES Frontend
- MES API
- Redis
- MES DB

## Sequence diagram: чтение списка заказов

```mermaid
sequenceDiagram
    actor Operator as Оператор
    participant FE as MES Frontend
    participant API as MES API
    participant Cache as Redis
    participant DB as MES DB

    Operator->>FE: Открывает страницу заказов
    FE->>API: GET /orders?status=NEW&page=1&sort=created_desc
    API->>Cache: GET orders:list:NEW:1:created_desc

    alt Cache hit
        Cache-->>API: Список заказов
        API-->>FE: 200 OK + данные из кеша
        FE-->>Operator: Быстрое отображение списка
    else Cache miss
        Cache-->>API: Нет данных
        API->>DB: SELECT список заказов
        DB-->>API: Данные списка
        API->>Cache: SET orders:list:NEW:1:created_desc TTL=30s
        API-->>FE: 200 OK + данные из БД
        FE-->>Operator: Отображение списка
    end
````

## Sequence diagram: изменение статуса заказа

```mermaid
sequenceDiagram
    actor Operator as Оператор
    participant FE as MES Frontend
    participant API as MES API
    participant DB as MES DB
    participant Cache as Redis

    Operator->>FE: Меняет статус заказа
    FE->>API: PATCH /orders/{id}/status
    API->>DB: UPDATE order status
    DB-->>API: OK

    API->>Cache: DEL order:card:{id}
    API->>Cache: DEL orders:stats
    API->>Cache: DEL orders:list:*
    API-->>FE: 200 OK
    FE-->>Operator: Статус обновлён
```

## Стратегия инвалидации кеша

Я предлагаю использовать комбинированную стратегию инвалидации:

### 1. Инвалидация по ключу

После изменения статуса заказа нужно:
- удалить кеш карточки заказа
- удалить кеш агрегатов
- удалить кеш списков заказов, которые стали неактуальными

Это основной способ сделать так, чтобы после записи пользователи не получали устаревшие данные.

### 2. Временная инвалидация (TTL)

Дополнительно для ключей нужно задать небольшой TTL:
- список заказов: 15–30 секунд
- агрегаты по статусам: 10–30 секунд
- карточка заказа: 30–60 секунд

TTL нужен как дополнительная страховка на случай, если программная инвалидация не сработала или не покрыла все ключи.

### Почему подходит именно такая стратегия

Эта стратегия подходит, потому что:
- операторам важны достаточно свежие данные
- слишком длинный кеш здесь опасен
- только TTL без удаления ключей приведёт к устаревшим данным
- только программная инвалидация без TTL сделает систему более хрупкой.

## Сравнение стратегий

| Стратегия                     | Плюсы                                                                                                                                                   | Минусы                                                                                                      | Вывод                                                          |
| ----------------------------- |---------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------| -------------------------------------------------------------- |
| Cache-Aside + удаление по ключу + TTL | Простая реализация, хорошо подходит для сценариев с частым чтением; даёт контроль над актуальностью данных; позволяет кешировать только горячие запросы | Нужно продумать схему ключей и инвалидацию                                                                  | Лучший вариант для MES                                         |
| Только TTL                    | Очень просто внедрить, не нужно писать сложную логику обновления                                                                                        | Данные могут устаревать до истечения TTL, нет мгновенной реакции на изменение статуса                       | Можно использовать только как дополнительную меру, но не как основную |
| Write-Through                 | Кеш обновляется сразу при записи, может быть удобен для простых объектов                                                                                | Плохо подходит для списков и агрегатов, усложняет запись, не решает основную проблему чтения так эффективно | Для данного кейса хуже, чем Cache-Aside                        |
| Refresh-Ahead                 | Может ускорять работу с очень горячими ключами, помогает держать кеш прогретым                                                                          | Сложнее в реализации, трудно предсказать все нужные ключи; можно обновлять невостребованные данные          | Можно рассматривать позже как дополнительную оптимизацию       |
